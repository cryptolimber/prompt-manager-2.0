# prompt_manager_2.0 文档

## 一、背景

当前需要开发一个用于管理 Prompt 的多环境桌面端工具。

系统当前只支持三个固定环境类型：

1. 开发环境
2. 测试环境
3. 预发布环境

三个环境均使用 MySQL 数据库，并且只操作同一张表：

`aiag_prompt_template`

该工具的核心目标是：

1. 在开发环境中查看、编辑和维护 Prompt 数据。
2. 将开发环境中的指定 Prompt 记录推送到测试环境。
3. 将开发环境中的指定 Prompt 记录推送到预发布环境。
4. 推送时只更新用户指定的数据条目，不做全量覆盖。

## 二、技术栈要求

该桌面端工具使用以下技术栈实现：

1. Python：作为主要开发语言。
2. PyMySQL：用于连接和操作远程 MySQL 数据库。
3. Peewee：作为 ORM 框架，用于封装本地 SQLite 数据库和远程 MySQL 数据库的表模型及数据操作逻辑。
4. NiceGUI：用于实现桌面端可视化界面。

开发过程中需要注意工程可维护性，要求如下：

1. 同类职责代码需要归类管理，避免所有逻辑堆叠在单个文件中。
2. 数据库连接、数据库模型、数据访问逻辑、业务逻辑、界面逻辑需要尽量拆分。
3. MySQL 远程数据库相关代码和 SQLite 本地状态维护代码需要区分管理。
4. NiceGUI 页面代码只负责界面展示和用户交互，不应直接堆叠大量数据库操作逻辑。
5. 数据同步、Prompt 编辑、环境配置管理等核心业务逻辑应封装为独立服务模块。
6. 配置文件读写逻辑应单独封装，便于后续维护和扩展。
7. 代码结构需要清晰，便于后续新增环境配置、同步策略、推送策略和界面功能。

## 三、aiag_prompt_template 表结构

该表对应的 ORM Model 原始结构如下：

```python
class AiagPromptTemplate(Base):
    __tablename__ = 'aiag_prompt_template'

    id = Column(BigInteger, primary_key=True, comment='id;ID')
    prompt_content = Column(Text, comment='提示词内容;完整的提示词文本')
    code = Column(String(100), comment='提示词编码;枚举值，对应代码中的标识')
    version = Column(String(50), comment='版本号;例如1,2,3')
    description = Column(String(1024), comment='功能描述;提示词的作用和使用场景')
    category = Column(String(50), comment='提示词分类;提示词的分类，如教学、助手、创作等')
    is_active = Column(Integer, comment='是否启用;1-启用,0-停用')
    tenant_id = Column(BigInteger, comment='租户ID;租户ID')
    org_id = Column(BigInteger, comment='机构ID;机构ID')
    create_time = Column(DateTime, default=datetime.now, comment='创建时间;创建时间')
    status = Column(Integer, comment='状态;0-逻辑删除,1-新增,2-修改')
```

实际开发时，需要基于上述字段使用 Peewee 定义对应的远程 MySQL 表模型。

## 四、桌面端软件功能要求

### 1. 数据库环境配置

软件需要支持数据库环境的配置管理，包括：

1. 添加新的数据库环境。
2. 修改已有数据库环境配置。
3. 删除已有数据库环境配置。
4. 每个环境都支持自定义环境名称，便于用户识别。

当前系统只支持三个固定环境类型：

1. 开发环境
2. 测试环境
3. 预发布环境

添加或修改数据库环境时，用户必须选择一个环境类型。

环境类型只能从以下三个选项中选择：

1. 开发环境
2. 测试环境
3. 预发布环境

添加或修改数据库环境时，需要用户手动输入以下字段：

1. 环境类型
2. 环境名称
3. 主机 IP
4. 端口号
5. 用户名
6. 密码，明文输入即可
7. 数据库名称

其中：

1. 环境类型用于标识该数据库配置属于开发环境、测试环境还是预发布环境。
2. 环境名称用于给用户展示，便于用户在界面中识别不同数据库配置。
3. 环境类型是固定枚举值，不支持用户自定义新增其他环境类型。

上述数据库配置信息需要持久化到本地配置文件中。

配置文件要求如下：

1. 默认文件名：`PROMPT_MANAGER_CONFIG.json`
2. 默认存储位置：当前用户的用户目录
3. Windows 示例路径：`C:\Users\username`
4. 数据库配置页面需要显示该配置文件的实际存储路径

### 2. 提示词管理面板

提示词管理面板需要支持从已配置的数据库环境中查看 Prompt 数据。

功能要求如下：

1. 页面提供数据库环境下拉框。
2. 用户选择某个数据库环境后，系统连接对应数据库。
3. 系统查询该环境下 `aiag_prompt_template` 表的数据。
4. 查询结果以表格形式展示。
5. 表格需要支持分页，避免一次性加载过多数据影响渲染性能。

分页要求如下：

1. 默认每页显示 15 条数据。
2. 支持用户选择每页显示数量。
3. 可选值为：10、15、20。
4. 支持上一页、下一页操作。

表格默认展示以下字段：

1. `id`
2. `code`
3. `version`
4. `description`
5. `category`
6. `is_active`

### 3. 单条 Prompt 详情与编辑

提示词管理表格中，每条记录最右侧需要提供两个操作按钮：

1. 详情
2. 编辑

#### 详情功能

点击详情按钮后，需要展示该条记录的全部字段信息，包括：

1. `id`
2. `prompt_content`
3. `code`
4. `version`
5. `description`
6. `category`
7. `is_active`
8. `tenant_id`
9. `org_id`
10. `create_time`
11. `status`

#### 编辑功能

点击编辑按钮后，需要支持用户修改该条 Prompt 记录的字段内容。

编辑完成后，修改应写回当前选中的远程数据库环境中的 `aiag_prompt_template` 表。

### 4. 本地修改状态维护

由于 `AiagPromptTemplate` ORM Model 中没有定义 `update_time` 字段，因此软件需要在本地维护 Prompt 记录的修改状态。

本地状态维护使用 SQLite 数据库实现，不修改远程 MySQL 数据库表结构，也不影响远程数据库原有字段。

#### 4.1 本地 SQLite 表

本地 SQLite 中需要创建一张状态维护表：

`aiag_prompt_local_status_maintain_tb`

该表用于记录远程 Prompt 数据在本地的同步状态和修改状态。

#### 4.2 同步触发时机

当用户添加一个数据库配置后，系统需要测试该数据库是否可以正常连接。

如果连接成功，则需要从该数据库的 `aiag_prompt_template` 表中同步全部 Prompt 记录到本地 SQLite 状态表中。

#### 4.3 本地状态表需要记录的内容

同步时，本地状态表需要记录远程表中的以下信息：

1. 远程 Prompt 记录 ID
2. 远程 Prompt 记录创建时间
3. 本地最后修改时间
4. 数据库唯一标识
5. 环境类型
6. 删除状态标记

其中：

1. `last_modify_time`：记录本地最后修改时间，格式精确到年月日、时分秒。
2. `db_identify`：用于唯一标识一个远程数据库环境下的目标表。
3. `env_type`：用于标识该记录来源于开发环境、测试环境还是预发布环境。
4. `delete`：软删除字段，用于标记本地状态记录是否已失效。

#### 4.4 db_identify 生成规则

`db_identify` 使用以下格式生成：

`数据库IP:端口号:表名`

示例：

`127.0.0.1:3306:aiag_prompt_template`

这样可以避免用户删除数据库配置后，又重新录入相同数据库配置时，导致本地状态数据重复同步。

#### 4.5 删除数据库配置时的处理逻辑

当用户删除某个数据库配置时，不需要立即删除该数据库对应的本地 SQLite 状态记录。

处理规则如下：

1. 找到该数据库配置对应的 `db_identify`。
2. 将本地 SQLite 状态表中属于该 `db_identify` 的记录全部标记为禁用或软删除。
3. 不影响远程 MySQL 数据库中的 `aiag_prompt_template` 数据。

#### 4.6 删除 Prompt 记录时的处理逻辑

当用户在桌面端可视化删除某条 Prompt 记录时，本地 SQLite 状态表中对应的记录也需要删除。

处理规则如下：

1. 如果是删除数据库配置，只对本地状态表做软删除或禁用处理。
2. 如果是删除某条具体的 Prompt 记录，则本地状态表中对应记录执行硬删除。
